After updating CachyOS (`linux-cachyos 7.1.3-2`), the system became unbootable through the normal Limine boot entry. Selecting CachyOS from the firmware menu caused an immediate reboot with no kernel panic or error message.

This guide documents the complete troubleshooting process, the commands used, the investigation performed, and the conclusions reached.

<!--more-->

## Symptoms

Following a full system update:

- Windows Boot Manager continued to boot normally.
- Selecting **CachyOS** opened Limine.
- Choosing the Linux entry immediately rebooted the machine.
- No kernel panic or emergency shell appeared.
- Selecting **EFI Fallback (UEFI OS)** booted CachyOS successfully.

This suggested the installation itself might still be intact.

* * *

## Boot into a Live USB

Boot the latest CachyOS installation media and mount the installed system.

Identify the partitions:

```bash
lsblk -f
```

Example layout:

| Partition | Filesystem | Purpose |
|-----------|------------|---------|
| `/dev/nvme1n1p5` | FAT32 | EFI System Partition |
| `/dev/nvme1n1p6` | Btrfs | Root Filesystem |

Mount the installation:

```bash
sudo mount -o subvol=@ /dev/nvme1n1p6 /mnt
sudo mount /dev/nvme1n1p5 /mnt/boot
sudo arch-chroot /mnt
```

* * *

## Verify Kernel Installation

Check that the installed kernel packages still exist.

```bash
pacman -Q linux-cachyos linux-cachyos-headers
```

Output:

```
linux-cachyos 7.1.3-2
linux-cachyos-headers 7.1.3-2
```

The kernel packages were installed correctly.

* * *

## Verify the Bootloader

Since CachyOS was installed with Limine:

```bash
bootctl status
```

Result:

```
Systemd-boot not installed.
```

This is expected and confirms the system is not using systemd-boot.

* * *

## Inspect `/boot`

Initially:

```bash
ls -lh /boot
```

returned:

```
EFI/
loader/
limine.conf
limine.conf.old
amd-ucode.img
```

At first glance it appeared that the kernel was missing.

* * *

## Rebuild the Initramfs

Regenerate every installed kernel.

```bash
mkinitcpio -P
```

Output showed:

```
Initramfs stored in

/boot/<machine-id>/linux-cachyos/
```

This confirmed that CachyOS is using **Unified Kernel Images (UKIs)** rather than storing kernels directly in `/boot`.

* * *

## Verify Generated UKIs

List every boot file.

```bash
find /boot -type f
```

Relevant files:

```
linux-cachyos/vmlinuz-linux-cachyos
linux-cachyos/initramfs-linux-cachyos

linux-cachyos-lts/vmlinuz-linux-cachyos-lts
linux-cachyos-lts/initramfs-linux-cachyos-lts
```

This confirmed:

- Kernel exists
- Initramfs exists
- UKIs were generated correctly

The issue was therefore **not** a missing kernel.

* * *

## Check Boot Behaviour

Testing every firmware boot option produced the following results.

| Boot Entry | Result |
|------------|--------|
| Limine | Immediate reboot |
| Windows Boot Manager | Boots normally |
| EFI Fallback (UEFI OS) | Boots CachyOS successfully |

This ruled out:

- filesystem corruption
- broken kernel
- broken initramfs
- damaged UKIs

* * *

## Inspect EFI Boot Entries

Display all firmware boot entries.

```bash
sudo efibootmgr -v
```

Relevant entries:

```
Boot000B
\EFI\LIMINE\LIMINE_X64.EFI

Boot000C
\EFI\BOOT\BOOTX64.EFI
```

Observed behaviour:

| Entry | Result |
|-------|--------|
| `LIMINE_X64.EFI` | Reboots immediately |
| `BOOTX64.EFI` | Boots successfully |

This strongly suggested that the problem was isolated to the primary Limine EFI executable.

* * *

## Compare EFI Executables

Compute SHA-256 hashes.

```bash
sha256sum /boot/EFI/BOOT/BOOTX64.EFI
sha256sum /boot/EFI/LIMINE/LIMINE_X64.EFI
```

Output:

```
BOOTX64.EFI

0e4fee4c294d4e82eb01db531974691e8b4a0e9c89f0d7dc107af519575b3d62

LIMINE_X64.EFI

37aca59412c36333f775091c3398d49eb0c4b536690fe63e6338321405cf7a71
```

The binaries are different.

Because only one of them boots successfully, the EFI executable itself became the primary suspect.

* * *

## Verify Installed Limine Version

```bash
pacman -Q limine
```

Output:

```
limine 12.5.0-1
```

* * *

## Temporary Workaround

Since the fallback loader works correctly, configure the firmware to boot it first.

```bash
sudo efibootmgr -o 000C,0000
```

This changes the boot order to:

1. EFI Fallback (`BOOTX64.EFI`)
2. Windows Boot Manager

allowing the machine to boot normally while further investigation continues.

* * *

## Further Investigation

Compare both EFI executables.

```bash
file /boot/EFI/BOOT/BOOTX64.EFI
file /boot/EFI/LIMINE/LIMINE_X64.EFI
```

Inspect the directories.

```bash
ls -l /boot/EFI/BOOT
ls -l /boot/EFI/LIMINE
```

If confirmed safe, replace the broken executable with the working fallback copy.

```bash
sudo cp /boot/EFI/BOOT/BOOTX64.EFI \
/boot/EFI/LIMINE/LIMINE_X64.EFI
```

Then reboot and test the normal Limine firmware entry again.

* * *

## Conclusions

The investigation established that:

- The Btrfs filesystem is healthy.
- The EFI partition is healthy.
- The kernel packages are installed.
- UKIs are generated successfully.
- The Linux installation is fully bootable.
- The failure only occurs when booting through `EFI/LIMINE/LIMINE_X64.EFI`.

This strongly indicates that the issue lies with either:

- a corrupted `LIMINE_X64.EFI`,
- a Limine packaging regression,
- or a firmware compatibility issue affecting the primary Limine EFI executable.
